# PyarelalAI Backend

This repository contains the backend service for PyarelalAI. It is a TypeScript + Express application that provides authentication, user credit management, AI-assisted project creation, real-time WebSocket updates, and background job processing for code generation.

The backend is built around:
- Express.js for REST APIs
- Drizzle ORM with PostgreSQL
- Redis + BullMQ for background workers
- Socket.IO for live project updates
- Zod for request validation
- TypeScript with NodeNext module settings

---

## 1. Overview

The backend powers the core product workflow:

1. Users sign up or sign in
2. JWT-based authentication is issued via an HTTP-only cookie
3. Users can create projects with an initial prompt
4. Project creation consumes credits and queues an AI generation task
5. A background worker processes the task and streams generated HTML back to the frontend through WebSockets

This service is designed for a modern full-stack experience where the UI can receive generation status and code chunks in real time.

## 3. Project Structure

```text
backend/
├── docker-compose.yml         # Local PostgreSQL container setup
├── drizzle.config.ts          # Drizzle configuration
├── package.json               # Scripts and dependency versions
├── tsconfig.json              # TypeScript compiler settings
├── drizzle-migrations/        # SQL migration files generated by Drizzle
├── src/
│   ├── server.ts              # Main server entry point
│   ├── controllers/           # API route handlers
│   ├── routes/                # Express route definitions
│   ├── services/              # Business logic
│   ├── middlewares/           # Auth middleware and request guards
│   ├── models/                # Drizzle schema definitions
│   ├── lib/                   # Database, queue, socket, and migration helpers
│   ├── utils/                 # Token and helper utilities
│   ├── validations/           # Zod validation schemas
│   └── workers/               # Background job workers
```

### Main folders explained

- controllers/: Contains request/response logic for auth, user, and project flows.
- services/: Contains reusable business logic such as project creation and credit enforcement.
- models/: Defines database tables using Drizzle schema objects.
- lib/: Shared infrastructure helpers for PostgreSQL, Redis/BullMQ, WebSockets, and migrations.
- middlewares/: Auth protection for secured routes.
- workers/: Background processing for long-running AI generation jobs.
- validations/: Request payload validation with Zod.

## 4. Installation

### Prerequisites
- Node.js and npm installed

- Redis running locally or accessible via URL

### Steps

```bash
cd backend
npm install
```

If you want a local PostgreSQL database:

```bash
docker compose up -d
```

This starts a PostgreSQL container using the configuration from docker-compose.yml.

## 5. Environment Variables

Create a .env file in the backend folder with the following values:

```env
PORT=5000
NODE_ENV=development
DATABASE_URL=postgres://myuser:mypassword@localhost:5432/pyaradb
TRUSTED_ORIGINS=http://localhost:5173
JWT_ACCESS_SECRET=your_jwt_secret
REDIS_URL=redis://127.0.0.1:6379
GEMINI_API_KEY=your_gemini_api_key
```

### Variable descriptions
- PORT: Port for the backend server
- NODE_ENV: Environment mode
- DATABASE_URL: PostgreSQL connection string
- TRUSTED_ORIGINS: Allowed frontend origins for CORS
- JWT_ACCESS_SECRET: Secret used to sign authentication tokens
- REDIS_URL: Redis connection string for BullMQ
- GEMINI_API_KEY: Google Gemini API key used by the code generation worker

> Note: The project currently uses a local Postgres setup by default, but you can switch to Neon or any PostgreSQL-compatible provider by changing DATABASE_URL.

## 7. Database Model Overview

The backend uses Drizzle ORM schemas stored in src/models.

### Users
The users table stores:
- id
- email
- password
- name
- credits
- totalCreations
- createdAt
- updatedAt

### Website Projects
The website_projects table stores:
- id
- name
- initialPrompt
- currentCode
- currentVersionIndex
- isPublished
- userId
- createdAt
- updatedAt

These models are connected through the userId foreign key.

## 8. API Endpoints

The backend exposes the following main routes.

### Authentication
- POST /api/auth/sign-up
- POST /api/auth/sign-in
- POST /api/auth/sign-out

### User
- GET /api/user/credits

### Projects
- POST /api/project/create-project

### Auth protection
Protected routes use the protectRoute middleware, which checks the auth_session cookie and verifies the JWT.

## 9. Request Flow Examples

### Sign up
```bash
curl -X POST http://localhost:5000/api/auth/sign-up \
  -H "Content-Type: application/json" \
  -d '{
    "email": "user@example.com",
    "password": "password123",
    "name": "John Doe"
  }'
```

### Create a project
```bash
curl -X POST http://localhost:5000/api/project/create-project \
  -H "Content-Type: application/json" \
  -b "auth_session=<token>" \
  -d '{
    "name": "Landing Page",
    "initialPrompt": "Create a modern startup landing page with hero section and pricing"
  }'
```

## 10. Background Jobs and Real-Time Generation

When a project is created:
1. The app validates the payload
2. It checks whether the user has sufficient credits
3. It deducts credits and creates a project record
4. It pushes a job to the BullMQ queue
5. The worker processes the job and uses Gemini to generate HTML code
6. The generated code is streamed over Socket.IO and saved to the database

This enables the frontend to receive status updates such as:
- refining_prompt
- streaming_code
- completed
- failed

---

## 11. WebSocket Setup

The backend initializes a Socket.IO server on the main HTTP server.

Clients can join a project-specific room by emitting:

```javascript
socket.emit("join_project_canvas", projectId);
```

The server then broadcasts:
- generation_status
- code_stream_chunk
